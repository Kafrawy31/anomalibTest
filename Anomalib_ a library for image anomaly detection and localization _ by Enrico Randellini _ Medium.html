Anomalib: a library for image anomaly detection and localization

With Anomalib at hands, we can manage the images of a custom dataset, fine-tune state of the art pretrained models and test their ability to find abnormal images and localize the corresponding anomalous areas

The introduction of the library Anomalib says “Anomalib is a deep learning library that aims to collect state-of-the-art anomaly detection algorithms for benchmarking on both public and private datasets. Anomalib provides several ready-to-use implementations of anomaly detection algorithms described in the recent literature...”.

Personally, I find Anomalib a very powerful library to deal with tasks related to image anomaly detection and localization. The data management as well as the use of different state of the art models is wrapped by easy to use functions and APIs. Sometimes, however, the documentation is jagged and unclear. In this post I want to take a journey into the image anomaly detection world analyzing every steps and all interesting details of the library, from the custom dataset building to the trained model serving.

Just some remarks

  • Anomalies are not standard data. But what is a standard data? Usually depends on the context. Thus, if we want to check if the final product of a production pipeline is free of defects, standard data are images of the product in perfect conditions, without defects. Anomalous images are those with some defects and, if the pipeline works well, the defects are rare and even small in size compared to those of the entire image.
  • Usually, we don’t know what anomalous data looks like. A defect can manifest itself in any part and in any form.
  • Supervised approaches are generally ineffective for this type of prob-
    lem due to a lack of sufficient representative samples in the anomalous class.
  • Roughly speaking, anomaly detection algorithms are trained in an unsupervised manner. During the training steps they become aware of how normal data are made and distributed. Only in the validation and testing phase they are used to recognize anomalous examples by comparing against the learned distribution of normal data.
  • This unsupervised nature of anomaly detection makes it well suited for real-world applications such as industrial, medical and security, where a
    clearly defined anomalous class is often lacking.
  • Anomalous images can be detected and the corresponding anomalous areas can be localized. With Anomalib, to train a model to detect anomalous images it is sufficient havig a dataset with normal and abnormal images, where the last ones are used just for validate and test the model. At inference time, the model gives a probability score to classify an image as normal or abnormal. Another output of the model is the heatmap of the image, where the “hot” areas correspond to a high level of abnormality. The quality of the heatmap is increased if the dataset comes with a segmentation mask of the anomalous area for each anomalous image. In this case the validation process also involves a segmentation phase which improves the localization of the anomaly.
  • In this discussion we will consider a dataset without the mask of the anomalous areas and train the models to classify an image as normal or abnormal. We will also check the quality of the heatmap to localize the anomalous areas.

You can find the original code and deeper detailes on my Github repository at this link.

1. Build a custom dataset

Anomalib allows us to use benchmarking dataset as MVTec using specially dedicated APIs. On the other hand, not every datasets have the structure of MVTec and not every datasets have the anomalous images with the anomalous area appropriately segmented.

In order to take confidence with the library, I built a toy custom dataset by simplifying the specifications of a real industrial project. The problem is the following. There is an industrial machine with many components. A sequence of actions is performed on each component, changing their arrangement from time to time. The request is to verify that, at the end of each step, the arrangement of the components was the desired one.

To schematize the machine and its components, I took a wooden box and four rectangular Lego pieces placing them at the corners of a rectangle. The initial configuration includes the Lego pieces all arranged horizontally. The first action consists in turning the piece on the top right counterclockwise by 90 degrees, so that from horizontal it is arranged vertically. Thus, the top left is turning counterclockwise by 90 degrees, thus the down left and, finally, the down right. As shown in Figure 1, each of the final state is the correct configuration at the end of each step.

Any other configuration different from the right one is to be considered an abnormal configuration. Actually, this means that the action on the component was not performed correctly. For example, as shown in Figure 2, if at the end of the first step, the top right Lego piece is not completely rotated of 90 degrees, then the corresponding image is an anomaly image, and the anomaly is located in correspondence of the not well rotated Lego piece. For this application I chose as anomalous images, configurations where the Lego piece is rotated of 30, 45 and 60 degrees.

To build a dataset, for each correct and abnormal configuration I recorded a video with my smartphone, using a frame rate of 30 fps and a resolution of 1920x1080 pixels. During recording, I moved the camera to shoot the scene from multiple points of view, even not optimal ones, in order to introduce noise as happens in real cases. On average, I shot the correct configurations for about 120 seconds, while the anomalous ones for about 30 seconds. Furthermore, in order to introduce other noise, I placed the wooden box on a carpet with many colored shapes so that, by moving the camera, the shapes in background also varied.

With the following script we can extract the frames from the corresponding videos, thus for each different configuration we collect the images for the standard and anomalous cases. I center crop each image and resize it to 256x256 pixels.

2. Data loading

In order to use the Anomalib APIs to load the data and then train and test the models on our custom dataset, the folder tree must be structured as shown in Figure 3. For each folder of each category I must have a subdirectory corresponding to the normal images, i.e. the 90_DEG one. At the same level of the 90_DEG folder, there must be another subdirectory called “abnormal”. In the abnormal subdir there must be as many folders as anomaly types, i.e. 30_DEG, 45_DEG and 60_DEG, each one with the corresponding anomalous images.

To manage the data of a custom dataset, in particular to split it in a train, test and validation subset, we can use the Folder class as follows. I will focus on the one_up category, but the same things also apply to the other categories.

The attributes of the Folder object have to be filled as follows:

  • name: the name of the category, i.e. the name of the folder where are stored the normal and abnormal images
  • root: the root of the directory of the dataset, namely the root to the “one_up” directory
  • normal_dir: the name of the folder where are stored the normal images, in this case “90_DEG”
  • abnormal_dir: the name of the folder where are stored the abnormal images, in this case “abnormal”
  • task: we use CLASSIFICATION because the datset have just normal and abnormal images. If we also have the segmentation mask for each anomlous image we can use SEGMENTATION
  • seed: this is set to repeat each time the same train, val and test split
  • normal_split_ratio: ratio to split the normal training images and add to the test set in case the test set doesn’t contain any normal images. By default is set to 0.2
  • val_split_mode: determines how the validation dataset is obtained. By default is set to FROM_TEST
  • val_split_ratio: fraction of the train or test images that will be reserved for validation. The images are taken from the train or test set depending on the val_split_mode. By default 0.5
  • train_batch_size and eval_batch_size: set the value of the bach of images used to train and validate the model. By default are set to 32
  • image_size: tuple to set the width and heigth of the image. Here is commented because we already resized the images to (256,256) pixels

Once the datamodule Folder object has been instatiated, we can exctract the train, validation and test dataloader to itarete over the batch of images. The images are automatically transformed to torch tensors normalized between 0 and 1. We can also extract the dataframes corresponding to the train, validation and test splits, check some statistics about the distribution of normal and abnormal images in the different datasets ad save them as .csv files.

3. Model training and testing

In this post I will not focus on the details and architectures of the models. From this page of the Anomalib documentation, we can find all the models implemented in the library and their original papers. In the following I want to focus on the best model I found for my custom dataset, namely the Reverse Distillation one, and show how it is implemented, trained and tested. In my Github repository you can also find how to train and test other models availables in Anomalib.

First, I instantiated the model. Then I set it up the ModelCheckpoint and the EarlyStopping objects in the callbacks list. With the ModelCheckpoint you decide how to save the checkpoint during the validation phase. Here I have set the condition that the validation step is executed at the end of each epoch. Thus, if the AUROC value is greater than that of the previous step, then I override the checkpoint. With the EarlyStopping you decide how to stop the training process. Here I have set the condition that the stop depends on the trend of the AUROC metric. If it no longer grows beyond a certain number of epochs equal to patience, then the training is finished.

Then I set up the logger and the Engine object. The training starts using the fit function of the Engine, having as arguments the datamodule and the model previously instantiated. It should be remembered that for this discussion the datamodule is set to read the data relating to the one up configuration.

When the training starts, the logger prints some info about the model, in particular the number of the trainable parameters and the extimated dimension.

Once the training finished, I start the test phase using the test function of the Engine. In this case the Engine loads the test dataloader and applies the best model to the test images. At the end, the logger prints the AUROC and F1 score on the test set. In the last step, I export the weights of the best model in the torch format so that I can use it in the inference phase on a single image as discussed below.

The results are very impressive, namely the AUROC for the classification problem on the test set is 0.9999 and the F1 score is 0.9997. This is also confirmed by the confusion matrix on the test set in Figure 6.

The Engine save automatically the results of the test set in “/results/ReverseDistillation/one_up/v0/images”, where v0 stays for the first run. If we run once again the training phase, the Engine will create a new folder v1 end so on. The images folder has two subfolders called 90_DEG and abnormal, each one contains the images of the normal and abnormal classes of the test set. Each image is accompanied by the corresponding heatmap and the normal or abnormal scoring depending on the predicted classification as shown in Figures 7–10.

Although the performances for the binary classification problem are
almost perfect, the heatmaps show that the anomalous area are well localized around the Lego piece for the 45_DEG cases. For the cases 30_DEG and 60_DEG the predicted heatmaps are a tittle bit more confuse and the anomalous score is lower.

4. Model inference

It is interesting how to test the model just on a single image, that is what happened in an web application. With the TorchInferencer object I load the weights of the best model inthe torch format. Thus, once the image is loaded and preprocessed, I apply the predict function of the TorchInferencer to obtain the result object. This object has the following attributes:

  • pred_label: 0 for a normal image, 1 for an abnormal one
  • pred_score: the prediction score to be an anomalous image. Thus if the pred_label is 0, to find the pred_score to be normal I use 1-pred_score
  • image: the original image
  • heat_map: the heatmap for the test image. The hotter an area is, the higher the probability that it contains an anomaly
  • pred_mask: the black and white mask corresponding to the anomalous areas
  • segmentation: the segmented anomalous areas based on the extracted heatmap
